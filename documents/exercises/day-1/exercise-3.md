# Exercise 3 — Variables, Data Types, and Data Structures

**Estimated time:** 45–60 minutes

## Objective

Build a deep understanding of Terraform's variable system — primitive types, complex data structures, default values, validation, and precedence. By the end you will be writing professional, well-typed `variables.tf` files with meaningful descriptions.

## Part 1 — Primitive Variables (10 min)

### Step 1 — Create the project structure

```bash
mkdir ~/terraform-exercises/day1-exercise3
cd ~/terraform-exercises/day1-exercise3
touch main.tf variables.tf outputs.tf terraform.tfvars
```

### Step 2 — Declare primitive variables

Open `variables.tf` and add:

```hcl
variable "resource_group_name" {
  type        = string
  description = "Name of the Azure Resource Group."
}

variable "location" {
  type        = string
  description = "Azure region to deploy resources into."
  default     = "westeurope"
}

variable "instance_count" {
  type        = number
  description = "Number of instances to deploy."
  default     = 1

  validation {
    condition     = var.instance_count >= 1 && var.instance_count <= 10
    error_message = "instance_count must be between 1 and 10."
  }
}

variable "enable_diagnostics" {
  type        = bool
  description = "When true, diagnostic settings are deployed."
  default     = false
}
```

**Notice:**
- `type` enforces that the value matches the expected type. If someone passes `"two"` for `instance_count`, Terraform errors immediately.
- `description` is shown in `terraform plan` output and in documentation generated by tools like `terraform-docs`.
- `validation` blocks let you add custom constraints with human-readable error messages.

### Step 3 — Provide values in `terraform.tfvars`

```hcl
resource_group_name = "rg-variables-training"
location            = "westeurope"
instance_count      = 2
enable_diagnostics  = true
```

### Step 4 — Write `main.tf`

```hcl
terraform {
  required_version = ">= 1.3.0"
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 3.0"
    }
  }
}

provider "azurerm" {
  features {}
}

resource "azurerm_resource_group" "main" {
  name     = var.resource_group_name
  location = var.location
}
```

### Step 5 — Test validation

First, test the happy path:

```bash
terraform init
terraform plan
```

Now test validation failure — pass an out-of-range value:

```bash
terraform plan -var="instance_count=20"
```

You should see:
```
│ Error: Invalid value for variable
│
│   on variables.tf line X:
│   X: variable "instance_count" {
│
│ instance_count must be between 1 and 10.
```

This happens **before** Terraform contacts Azure — validation is evaluated client-side.

## Part 2 — Collection Types (15 min)

### Step 1 — Add list and map variables

Add these to `variables.tf`:

```hcl
variable "allowed_locations" {
  type        = list(string)
  description = "List of Azure regions this deployment is permitted in."
  default     = ["westeurope", "northeurope", "uksouth"]
}

variable "tags" {
  type        = map(string)
  description = "A map of tags to apply to all resources."
  default = {
    environment = "training"
    managed_by  = "terraform"
    team        = "platform"
  }
}
```

Update the resource group in `main.tf` to use the tags map:

```hcl
resource "azurerm_resource_group" "main" {
  name     = var.resource_group_name
  location = var.location
  tags     = var.tags
}
```

### Step 2 — Use a list value

Add a second validation block to check the location is in the allowed list:

```hcl
variable "location" {
  type        = string
  description = "Azure region to deploy resources into."
  default     = "westeurope"

  validation {
    condition     = contains(var.allowed_locations, var.location)
    error_message = "The location must be one of: ${join(", ", var.allowed_locations)}."
  }
}
```

> **Note:** The `allowed_locations` variable must appear **before** `location` in `variables.tf` for this cross-variable validation to work cleanly.

Try forcing a failure:

```bash
terraform plan -var="location=eastus"
```

Test with a valid region:

```bash
terraform plan -var="location=uksouth"
```

### Step 3 — Add an object variable for subnet configuration

Objects let you group related values into a single variable:

```hcl
variable "vnet_config" {
  type = object({
    name          = string
    address_space = string
    subnets       = list(object({
      name   = string
      prefix = string
    }))
  })
  description = "Virtual network configuration including subnets."
  default = {
    name          = "vnet-training"
    address_space = "10.0.0.0/16"
    subnets = [
      { name = "web",  prefix = "10.0.1.0/24" },
      { name = "data", prefix = "10.0.2.0/24" },
    ]
  }
}
```

Update `main.tf` to deploy the VNet and subnets:

```hcl
resource "azurerm_virtual_network" "main" {
  name                = var.vnet_config.name
  address_space       = [var.vnet_config.address_space]
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  tags                = var.tags
}

resource "azurerm_subnet" "main" {
  count                = length(var.vnet_config.subnets)
  name                 = var.vnet_config.subnets[count.index].name
  resource_group_name  = azurerm_resource_group.main.name
  virtual_network_name = azurerm_virtual_network.main.name
  address_prefixes     = [var.vnet_config.subnets[count.index].prefix]
}
```

Run plan and review how the object variable's nested fields are accessed with dot notation:

```bash
terraform plan
terraform apply -auto-approve
```

## Part 3 — Local Values (5 min)

Locals are computed values derived from variables or other expressions. They are not inputs — they cannot be overridden from outside.

Add a `locals.tf` file:

```hcl
locals {
  name_prefix = "${var.tags["environment"]}-${var.resource_group_name}"
  common_tags = merge(var.tags, {
    deployed_at = "2026-01"
    region      = var.location
  })
}
```

Update the resource group name to use the local:

```hcl
resource "azurerm_resource_group" "main" {
  name     = local.name_prefix
  location = var.location
  tags     = local.common_tags
}
```

Locals are extremely useful for:
- Computing derived names (e.g., `"rg-${var.env}-${var.project}"`)
- Merging tag maps from multiple sources
- Avoiding repetition of complex expressions

## Part 4 — Variable Precedence (10 min)

Terraform loads variable values in this order — **later sources override earlier ones:**

| Priority | Source |
|---|---|
| 1 (lowest) | Default values in `variable` blocks |
| 2 | `terraform.tfvars` |
| 3 | `*.auto.tfvars` (alphabetical order) |
| 4 | `-var-file="file.tfvars"` (left to right) |
| 5 | `-var="name=value"` CLI flags |
| 6 (highest) | `TF_VAR_name` environment variables |

### Step 1 — Test the CLI flag override

```bash
terraform plan -var="location=northeurope"
```

The plan should now show `northeurope` as the location, overriding the `terraform.tfvars` value.

### Step 2 — Test environment variable override

```bash
export TF_VAR_location="uksouth"
terraform plan
```

Notice `uksouth` takes effect. Unset after testing:

```bash
unset TF_VAR_location
```

### Step 3 — Test `.auto.tfvars`

Create `overrides.auto.tfvars`:

```hcl
location = "northeurope"
```

Run plan — you will see `northeurope` used. Then rename the file away:

```bash
mv overrides.auto.tfvars overrides.auto.tfvars.bak
```

## Part 5 — Outputs

Add `outputs.tf`:

```hcl
output "resource_group_name" {
  description = "The name of the deployed resource group."
  value       = azurerm_resource_group.main.name
}

output "resource_group_id" {
  description = "The Azure resource ID of the resource group."
  value       = azurerm_resource_group.main.id
}

output "vnet_id" {
  description = "The Azure resource ID of the virtual network."
  value       = azurerm_virtual_network.main.id
}

output "subnet_names" {
  description = "List of all deployed subnet names."
  value       = [for s in azurerm_subnet.main : s.name]
}
```

Apply and inspect outputs:

```bash
terraform apply -auto-approve
terraform output
terraform output -json
terraform output subnet_names
```

Outputs have three common uses:
1. **Human inspection** — paste the resource ID into the portal
2. **Inter-module communication** — a child module exposes outputs that the parent consumes
3. **CI/CD pipelines** — `terraform output -json` is parsed by downstream scripts

## Key Concepts Summary

| Concept | Description |
|---|---|
| `string`, `number`, `bool` | Primitive types |
| `list(T)` | Ordered sequence of T values |
| `map(T)` | Key-value map with string keys and T values |
| `object({ ... })` | Structured type with named attributes of different types |
| `validation` block | Custom constraint evaluated before plan |
| `locals` | Computed values that cannot be overridden externally |
| Variable precedence | Defaults → tfvars → auto.tfvars → `-var-file` → `-var` → `TF_VAR_` |
| `output` block | Exposes values after apply for inspection or module composition |
